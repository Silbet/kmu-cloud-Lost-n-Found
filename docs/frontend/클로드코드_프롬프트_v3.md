# 분실물 신고 및 보관소 연동 시스템 — 웹 프론트엔드 개발 프롬프트 (v3)

## 📌 프로젝트 개요

국민대학교 캠퍼스 내 **분실물 신고 및 보관소 연동 시스템**의 **웹 프론트엔드만** 개발한다. 백엔드와 AWS 인프라(EC2, DynamoDB, S3, SQS, Lambda 등)는 다른 팀원이 담당하므로, 이 프롬프트로 만드는 산출물은 **순수 React SPA**다.

### 디자인 원칙
- **꾸밈요소 없이 깔끔하고 베이직한 UI**
- **기능 동작이 최우선** — 화려한 애니메이션·그라데이션 금지
- **여백·타이포·정렬 중심의 미니멀 디자인**
- **단색 계열 + 1~2가지 포인트 컬러**만 사용
- 모든 화면에서 일관된 공통 컴포넌트 사용

### 핵심 작업 원칙: "프론트엔드만, 그러나 백엔드 연결은 완벽히 준비"
- **AWS 리소스를 직접 만들거나 호출하는 코드 작성 금지** (presigned URL 발급, SQS 직접 전송, DynamoDB SDK 등 절대 포함하지 않음)
- 백엔드 API의 **호출 인터페이스만 정의**하고, 실제 호출은 **Mock 어댑터로 대체**
- 환경변수 토글 하나(`VITE_USE_MOCK`)로 Mock ↔ 실제 API 전환 가능
- 백엔드 담당자가 받아 보고 **요청/응답 JSON 스키마, 엔드포인트 목록, 환경변수**만 맞추면 즉시 연결되도록 한다

## 🛠 기술 스택

- **프레임워크**: React 18 + Vite
- **언어**: TypeScript
- **라우팅**: React Router v6
- **상태 관리**: React Query (서버 상태 + 캐싱) + Zustand (로그인 정보·시스템 설정 같은 전역 클라이언트 상태)
- **스타일링**: Tailwind CSS (유틸리티 위주)
- **HTTP**: Axios
- **폼 / 검증**: React Hook Form + Zod

### 외부 라이브러리 최소화
- 차트는 운영자 대시보드에서만 가볍게 (Recharts 권장)
- QR 코드는 `qrcode.react` 사용
- 그 외 UI 라이브러리(MUI, Ant Design 등) 사용 금지 — Tailwind만으로 충분

## 🔌 백엔드 연동 추상화 (가장 중요)

### 환경변수 (`.env.example` 파일을 함께 생성)
```
VITE_API_BASE_URL=http://localhost:3000
VITE_USE_MOCK=true
VITE_AUTH_HEADER_NAME=Authorization
```

- `VITE_USE_MOCK=true` → 모든 API 호출이 `src/api/mock/` 의 Mock 어댑터로 라우팅 (localStorage 기반)
- `VITE_USE_MOCK=false` → 실제 백엔드 API 호출 (`VITE_API_BASE_URL` + 엔드포인트)

### API 모듈 구조 (`src/api/`)
- 각 도메인별 파일: `reports.ts`, `items.ts`, `matches.ts`, `pickups.ts`, `admin.ts`, `auth.ts`, `notifications.ts`
- 각 파일은 **동일한 함수 시그니처**를 가진 함수만 export
- 내부에서 `VITE_USE_MOCK` 체크 후 Mock 또는 실제 호출 분기

```ts
// 예시: src/api/reports.ts
export async function createLostReport(payload: CreateLostReportRequest): Promise<LostReport> {
  if (import.meta.env.VITE_USE_MOCK === 'true') return mockCreateLostReport(payload);
  return apiClient.post('/reports', payload).then(r => r.data);
}
```

### 이미지 업로드 추상화
```ts
// src/api/upload.ts
export async function uploadImage(file: File): Promise<{ imageUrl: string }> {
  if (import.meta.env.VITE_USE_MOCK === 'true') {
    return { imageUrl: await mockStoreImage(file) };
  }
  const formData = new FormData();
  formData.append('file', file);
  return apiClient.post('/uploads/image', formData).then(r => r.data);
}
```

### 인증 추상화
- Mock에서는 로그인 시 가짜 토큰 발급 → localStorage 저장 → Axios 인터셉터가 모든 요청 헤더에 자동 첨부
- 백엔드 연결 시 같은 인터페이스로 실제 토큰 받아 저장

## 👥 액터 (사용자 역할)

총 5종. 로그인 후 역할에 따라 메뉴와 화면이 달라진다.

| 액터 | 권한 | 비고 |
|---|---|---|
| **분실자** | 분실 신고 등록·수정·삭제·조회, 확인 요청, 수령 코드 확인, 본인의 수령 대기 취소 | 같은 학생이 습득자도 겸할 수 있음 |
| **습득자** | 습득물 등록·삭제(등록 상태 한정) | 같은 학생이 분실자도 겸할 수 있음 |
| **보관소 관리자** | 습득물 보관 위치 입력·변경, 상태 변경, 확인 요청 승인/반려, 수령 완료 처리, **찾기완료 → 종료 수동 전환**, 수령 대기 강제 취소, 폐기예정 전환 | |
| **운영 관리자** | 전체 목록 조회, 통계, 장기 미수령 모니터링, **시스템 설정 변경(N값 조정)** | **현재는 우리팀(개발자)가 담당**. 추후 다른 학교로 확장 시 전체 시스템 관리 역할 |
| **비로그인 사용자** | 분실물/습득물 목록 검색만 가능 (민감정보 마스킹 적용) | |

## 📊 도메인 타입 정의 (`src/types/`)

```ts
// types/report.ts
export type LostReportStatus = '접수' | '매칭후보있음' | '찾기완료' | '종료';

export interface LostReport {
  reportId: string;
  reporterId: string;
  reporterName: string;
  reporterContact: string;
  itemName: string;
  category: string;
  lostPlace: string;
  lostDate: string;      // ISO 8601
  description: string;
  status: LostReportStatus;
  createdAt: string;
  updatedAt: string;
}

// types/item.ts
export type FoundItemStatus = '등록' | '보관중' | '수령대기' | '수령완료' | '폐기예정';

export interface FoundItem {
  itemId: string;
  finderId: string;
  itemName: string;
  category: string;
  foundPlace: string;
  foundDate: string;
  description: string;
  imageUrl?: string;
  storageLocation?: string;
  status: FoundItemStatus;
  createdAt: string;
  updatedAt: string;
}

// types/match.ts
export type MatchStatus = '활성' | '확인요청중' | '승인' | '반려' | '비활성';

export interface Match {
  matchId: string;
  reportId: string;
  itemId: string;
  status: MatchStatus;
  rejectReason?: string;   // 반려된 경우 사유 (선택)
  createdAt: string;
  updatedAt: string;
}

// types/pickup.ts
export type PickupStatus = '수령대기' | '수령완료' | '취소';
export type CancelReason = '분실자취소' | '관리자취소' | '시스템자동취소';

export interface Pickup {
  pickupId: string;
  matchId: string;
  reportId: string;
  itemId: string;
  pickupCode: string;          // 6자리 숫자
  status: PickupStatus;
  waitingStartedAt: string;
  autoCancelAt: string;        // 자동 취소 예정 시각 (서버 계산값)
  completedAt?: string;
  cancelledAt?: string;
  cancelReason?: CancelReason;
}

// types/config.ts
export interface SystemConfig {
  longUnclaimedDays: number;     // 기본 30
  pickupAutoCancelDays: number;  // 기본 3
  matchDateRangeDays: number;    // 기본 7
}

// types/user.ts
export type UserRole = '분실자' | '습득자' | '보관소관리자' | '운영관리자';

export interface User {
  userId: string;
  email: string;
  name: string;
  contact: string;
  roles: UserRole[];   // 분실자/습득자는 같은 계정이 둘 다 가질 수 있어 배열로 관리
}

// types/notification.ts
export type NotificationType =
  | '매칭후보생성'      // 분실자에게: 새 매칭 후보 발견
  | '확인요청승인'      // 분실자에게: 확인 요청 승인됨
  | '확인요청반려'      // 분실자에게: 확인 요청 반려됨
  | '수령자동취소임박'  // 분실자에게: N-1일째에 임박 알림
  | '수령대기취소'      // 분실자에게: 수령 대기가 취소됨
  | '폐기검토필요';     // 보관소 관리자에게: 보관 N일 경과

export interface Notification {
  notificationId: string;
  userId: string;
  type: NotificationType;
  title: string;
  message: string;
  link?: string;        // 클릭 시 이동할 경로 (예: /pickups/:pickupId)
  isRead: boolean;
  createdAt: string;
}
```

## 🌐 API 엔드포인트 명세

### 인증
```
POST   /auth/signup          { email, password, name, contact, roles } → { user }
POST   /auth/login           { email, password } → { token, user }
POST   /auth/logout
GET    /auth/me              → User
```

### 분실 신고
```
POST   /reports              { itemName, category, lostPlace, lostDate, description, reporterContact } → LostReport
GET    /reports/my           → LostReport[]
GET    /reports/:reportId    → LostReport
PATCH  /reports/:reportId    → LostReport
                             # 백엔드 검증:
                             # - 상태가 접수/매칭후보있음일 때만 허용
                             # - 활성 확인 요청이 있으면 409 Conflict 반환 (수정 차단)
                             # - 성공 시 매칭 후보 재계산 자동 트리거
DELETE /reports/:reportId    # 백엔드 검증:
                             # - 상태가 접수/매칭후보있음일 때만 허용
                             # - CASCADE 매칭 후보 삭제
```

### 습득물
```
POST   /items                → FoundItem
GET    /items/my             → FoundItem[]
DELETE /items/:itemId        # 등록 상태에서만 가능 (백엔드 검증)
PATCH  /items/:itemId/storage  { storageLocation } → FoundItem   # 보관소 관리자
                             # 보관 위치 입력 시 등록 → 보관중 자동 전환 + 매칭 트리거
PATCH  /items/:itemId/status   { status } → FoundItem            # 보관소 관리자
GET    /items                → FoundItem[]                       # 보관소 관리자 전체 조회
```

### 이미지 업로드
```
POST   /uploads/image  (multipart/form-data: file) → { imageUrl }
```

### 매칭 / 확인 요청
```
GET    /reports/:reportId/matches  → Match[] (with 연결된 FoundItem 정보)
POST   /matches/:matchId/confirm   → Match  # 분실자가 확인 요청
                                    # 백엔드 검증: 활성 확인 요청 1건 제한
POST   /matches/:matchId/approve   → Match  # 보관소 관리자 승인 → Pickup 자동 생성, 수령 코드 발급
POST   /matches/:matchId/reject    { reason? } → Match
                                    # 매칭 상태는 '활성'으로 복귀 (재요청 가능)
                                    # 분실 신고 상태는 매칭후보있음 유지
```

### 검색
```
GET    /search/lost     ?category=&place=&dateFrom=&dateTo=&keyword=
GET    /search/found    ?category=&place=&dateFrom=&dateTo=&keyword=
                        # 비로그인 호출 가능
                        # 백엔드가 민감정보 마스킹 + 수령완료 제외
                        # 매칭 후보는 습득물이 '보관중' 이상일 때만 노출
```

### 수령
```
GET    /pickups/:pickupId           → Pickup
GET    /pickups/waiting             → Pickup[]   # 보관소 관리자, 수령 대기 목록
POST   /pickups/:pickupId/verify    { name, contact, code }
                                    → { allMatched: boolean, mismatches: ('name'|'contact'|'code')[] }
POST   /pickups/:pickupId/complete  → Pickup    # 위 verify 통과 후 호출
                                    # 호출 시 자동 처리:
                                    # - 습득물 → 수령완료
                                    # - 분실 신고 → 찾기완료
                                    # - 해당 습득물의 매칭 후보 비활성화
POST   /pickups/:pickupId/cancel    { reason: CancelReason } → Pickup
                                    # 분실자/보관소 관리자 호출 가능 (역할은 토큰에서 확인)
                                    # 호출 시 자동 처리:
                                    # - 습득물 → 보관중 복귀
                                    # - 분실 신고 → 매칭후보있음 복귀
                                    # - 매칭은 활성 유지 (재요청 가능)
```

### 분실 신고 종료 처리 (보관소 관리자)
```
POST   /reports/:reportId/finalize  → LostReport
                                    # 찾기완료 상태인 분실 신고를 종료로 전환
                                    # 분실자에게 물품을 실물 전달 완료한 시점에 보관소 관리자가 수동 호출
```

### 알림 (Notifications)
```
GET    /notifications              → Notification[]   # 내 알림 목록 (최신순)
GET    /notifications/unread-count → { count: number }
PATCH  /notifications/:id/read     → Notification
PATCH  /notifications/read-all     → { updated: number }
```

### 운영 관리자
```
GET    /admin/reports                     → LostReport[]
GET    /admin/items                       → FoundItem[]
GET    /admin/stats                       → { lostReportsByStatus, foundItemsByStatus, recentTrend }
GET    /admin/unclaimed                   → FoundItem[]  # 장기 미수령 = 폐기 검토 대기
GET    /admin/config                      → SystemConfig
PATCH  /admin/config                      → SystemConfig
```

### 응답 규약
- 성공: 2xx + JSON 본문
- 클라이언트 오류: 4xx + `{ error: { code, message } }`
- 충돌(비즈니스 룰 위반): 409 + `{ error: { code, message } }`
- 인증 실패: 401, 권한 부족: 403
- 서버 오류: 5xx

## 🔁 상태 자동 전환 정리 (중요)

여러 상태가 자동/수동으로 전환되므로 한 곳에서 명확히 정리한다. 프론트는 응답으로 받은 상태를 신뢰하고 표시만 하면 되지만, **사용자에게 어떤 액션이 어떤 결과를 일으키는지 안내**해야 한다.

### 분실 신고 상태 흐름
```
[접수] ───매칭 후보 생성됨(자동)───> [매칭후보있음]
[매칭후보있음] ───수령 완료됨(자동)───> [찾기완료]
[찾기완료] ───보관소 관리자가 수동 종료 처리───> [종료]
[매칭후보있음] ───수령 대기 취소됨(자동)───> [매칭후보있음] 유지
```

### 습득물 상태 흐름
```
[등록] ───보관 위치 입력됨(보관소 관리자)───> [보관중] (+ 매칭 트리거)
[보관중] ───확인 요청 승인됨───> [수령대기]
[수령대기] ───수령 완료됨───> [수령완료]
[수령대기] ───수령 대기 취소됨(자동/수동)───> [보관중]
[보관중/장기미수령] ───보관소 관리자가 수동 변경───> [폐기예정]
```

### 매칭 상태 흐름
```
[활성] ───분실자가 확인 요청───> [확인요청중]
[확인요청중] ───보관소 관리자 승인───> [승인] (Pickup 생성)
[확인요청중] ───보관소 관리자 반려───> [활성] (재요청 가능, rejectReason 기록)
[승인] ───수령 완료됨───> [비활성]
[승인] ───수령 대기 취소됨───> [활성] (재요청 가능)
[전체] ───분실 신고 수정 시 재계산───> [활성/비활성]
```

## 📱 화면별 상세 요구사항

### A. 공통

#### A-1. 로그인 / 회원가입
- 이메일 + 비밀번호 로그인
- 회원가입: 이름·연락처·역할(분실자/습득자/보관소관리자/운영관리자, 다중 선택 가능)
- 로그인 성공 시 역할에 맞는 기본 페이지로

#### A-2. 헤더
- 로고 + 역할별 메뉴
- **알림 종 아이콘** (로그인 시): 미읽음 알림 개수 배지 표시. 클릭 시 드롭다운 또는 `/notifications` 이동.
- 우측: 로그인 정보 또는 로그인 버튼
- 비로그인 사용자도 검색 메뉴는 항상 노출

#### A-3. 알림 목록 (`/notifications`)
- 최신순 카드 리스트
- 각 알림: 타입 아이콘 + 제목 + 메시지 + 시간 + (있을 시) 이동 링크
- 미읽음은 진한 배경, 읽음은 옅은 배경
- 상단에 [모두 읽음 표시] 버튼
- 알림 종류별 메시지 예시:
  - `매칭후보생성`: "새 매칭 후보가 X건 발견되었습니다."
  - `확인요청승인`: "확인 요청이 승인되었습니다. 수령 코드를 확인하세요."
  - `확인요청반려`: "확인 요청이 반려되었습니다. (사유: ...)"
  - `수령자동취소임박`: "수령 대기가 N시간 후 자동 취소됩니다. 보관소를 방문해주세요."
  - `수령대기취소`: "수령 대기가 취소되었습니다."
  - `폐기검토필요`: "보관 N일 경과 물품 X건이 폐기 검토 대상입니다."

#### A-4. 검색 페이지 (`/search`)
- 탭: 분실물 / 습득물
- 필터: 카테고리(드롭다운), 장소, 날짜 범위, 키워드
- 결과 카드 그리드: 이미지(있을 때), 물품명, 카테고리, 장소, 일시, 상태 배지
- **수령완료 습득물은 응답에 포함되지 않음** (백엔드 필터링)
- **습득물 매칭 후보는 '보관중' 이상부터 표시** (백엔드 필터링)
- **비로그인일 때 마스킹**:
  - 1차 책임은 백엔드 (마스킹된 값을 응답으로 내려줌)
  - Mock 모드에서는 **Mock 핸들러가 마스킹 처리**해서 일관성 유지
  - 프론트의 `utils/mask.ts`는 보조용 (백엔드가 마스킹을 안 했을 때의 안전망)
- 로그인 사용자: 매칭 가능성 있는 항목에서 [확인 요청] 버튼 노출

### B. 분실자

#### B-1. 분실 신고 작성 (`/reports/new`)
- 필드: 물품명, 카테고리(드롭다운), 분실 장소, 분실 일시(datetime-local), 상세 설명(textarea), 연락처(기본값 회원가입 정보)
- 모두 필수, 설명만 선택
- 제출 → 신고 완료 토스트 → 매칭 후보 페이지로 이동
- 매칭 결과: 있으면 표시, 없으면 "매칭 후보가 아직 없습니다. 새 매칭이 생기면 알림으로 알려드립니다."

#### B-2. 내 분실 신고 목록 (`/reports/my`)
- 테이블: 물품명 / 카테고리 / 분실 장소 / 분실 일시 / 상태 배지 / 매칭 후보 수 / 액션
- 최신 신고 순 정렬
- 상태 배지 색상은 디자인 섹션 참조

#### B-3. 분실 신고 상세 (`/reports/:reportId`)
- 신고 정보 + 매칭 후보 섹션

**수정 버튼 활성 조건 (3가지 모두 충족 시 활성)**:
1. 상태가 `접수` 또는 `매칭후보있음`
2. 해당 분실 신고에 **확인요청중**(`MatchStatus = '확인요청중'`) 매칭이 없음
3. 상태가 `찾기완료` 또는 `종료`가 아님

조건 불충족 시 버튼 비활성 + 정확한 이유 안내:
- 확인 요청 중일 때: **"확인 요청 진행 중에는 신고를 수정할 수 없습니다. 승인 또는 반려 후 다시 시도해주세요."**
- 찾기완료/종료 상태: **"이미 처리된 신고는 수정할 수 없습니다."**

수정 후 저장 시 안내 모달: **"수정사항이 저장되었습니다. 매칭 후보가 다시 계산됩니다. 기존 매칭 후보가 새 입력에 맞지 않으면 사라질 수 있습니다."**

**삭제 버튼 활성 조건**:
1. 상태가 `접수` 또는 `매칭후보있음`
2. 확인요청중 매칭이 없음

조건 불충족 시 안내 동일하게.

삭제 클릭 시 확인 모달: **"관련 매칭 후보도 함께 삭제됩니다. 연결된 습득물은 다시 '보관중' 상태로 돌아갑니다. 계속하시겠습니까?"**

**매칭 후보 섹션 — 반려된 매칭 표시**:
- 매칭 카드에 상태 배지 표시 (활성/확인요청중/승인/반려/비활성)
- **반려된 매칭**도 카드에 표시. 작은 정보 박스로 반려 사유 표시: "관리자 반려: {rejectReason}"
- 반려 후에도 [재확인 요청] 버튼 활성 (핫스팟 13 결정: 모든 취소/반려 케이스에서 재요청 가능)

#### B-4. 매칭 후보 / 확인 요청 (B-3 내부 섹션 또는 별도 영역)
- 카드 리스트: 물품명, 카테고리, 습득 장소, 습득 일시, 이미지
- 보관 위치는 표시하지 않음 (수령 단계에서만)
- **[확인 요청] 버튼**:
  - 분실 신고에 이미 활성 확인 요청이 있으면 **모든 매칭 후보의 버튼 비활성**
  - 안내: **"이미 진행 중인 확인 요청이 있습니다. 처리 후 다시 시도해주세요."**
  - 활성 요청 없으면 활성, 클릭 → 확인 모달 → POST
- 매칭 상태별 표시:
  - `활성` → [확인 요청] 버튼
  - `확인요청중` → "처리 대기 중" 텍스트
  - `승인` → "승인됨 — [수령 코드 보기]" 링크 (Pickup 페이지로)
  - `반려` → 반려 사유 표시 + [재확인 요청] 버튼
  - `비활성` → 카드 흐림 처리 + "수령 완료로 종료된 매칭" 표시

#### B-5. 수령 코드 화면 (`/pickups/:pickupId`)
- 확인 요청 승인 후 자동 생성된 **6자리 수령 코드** 크게 표시
- QR 코드 함께 표시
- 보관 위치 표시
- **자동 취소까지 카운트다운**: D-day 또는 시·분 단위 (서버 `autoCancelAt` 값 기준)
- 안내 문구:
  - "보관소 방문 시 **신분증**과 이 화면을 제시해주세요."
  - "**대리 수령은 불가합니다. 분실자 본인만 수령 가능합니다.**"
  - "수령 대기 N일이 경과하면 자동 취소됩니다." (N은 시스템 설정값 동적 표시)
- **[수령 대기 취소] 버튼** (분실자 본인용):
  - 클릭 시 확인 모달: **"수령 대기를 취소하면 해당 매칭이 다시 활성화되어 다른 후보를 확인 요청할 수 있습니다. 계속하시겠습니까?"**
  - 호출: `POST /pickups/:pickupId/cancel` with `reason: '분실자취소'`
  - 취소 성공 → 토스트 + 분실 신고 상세 페이지로 이동

### C. 습득자

#### C-1. 습득물 등록 (`/items/new`)
- 필드: 물품명, 카테고리, 습득 장소, 습득 일시, 상세 설명, 이미지(선택)
- 이미지 업로드: `uploadImage()` 호출 후 받은 `imageUrl`을 등록 payload에 포함
- 제출 후 안내: **"보관소에 물품을 전달해주세요. 보관 위치가 등록되면 매칭이 시작됩니다."**

#### C-2. 내 등록 목록 (`/items/my`)
- 테이블: 물품명 / 카테고리 / 습득 장소 / 습득 일시 / 상태 / 보관 위치 / 액션
- [삭제] 버튼 — `등록` 상태에서만 활성
- `보관중` 이후엔 비활성 + 안내: "보관 위치가 입력된 후에는 삭제할 수 없습니다."

---

### D. 보관소 관리자

#### D-1. 대시보드 (`/manager`)
- 요약 카드: 보관중 / 수령대기 / 확인요청 대기 / 장기 미수령 / **찾기완료 종료 대기** 각 개수
- 폐기 검토 알림 배지 (장기 미수령 개수)
- 빠른 진입 메뉴

#### D-2. 전체 습득물 관리 (`/manager/items`)
- 테이블: 물품명 / 카테고리 / 습득 일시 / 상태 / 보관 위치 / 액션
- 액션 (상태별로 다름):
  - 등록 → [보관 위치 입력] (입력 시 자동으로 `보관중` 전환 + 매칭 트리거)
  - 보관중 → [보관 위치 변경], [상태 변경]
  - 수령대기 → [수령 완료 처리]로 이동, [수령 대기 강제 취소] 버튼도 제공
  - 보관중/장기미수령 → [폐기예정으로 변경]
- 필터: 카테고리, 상태, 보관 위치, 날짜

**[수령 대기 강제 취소] (보관소 관리자용)**:
- 분실자가 연락 두절·취소 의사 표명 등의 사유로 관리자가 직접 취소할 때 사용
- 클릭 시 모달: 사유 입력(선택) → 확인
- 호출: `POST /pickups/:pickupId/cancel` with `reason: '관리자취소'`

#### D-3. 확인 요청 대기열 (`/manager/confirmations`)
- 각 확인 요청 = 카드. 분실 신고 정보 + 습득물 정보 나란히
- 관리자에게는 신고자 정보 풀 노출
- [승인] / [반려]
- 승인 → 백엔드가 자동으로 Pickup 생성·수령 코드 발급, 분실자에게 알림 발송
- 반려 → **사유 입력 모달** (선택이지만 권장):
  - "예: 사진과 실물이 다름, 분실 일자가 너무 차이남 등"
  - 백엔드에 사유 전달 → Match의 `rejectReason` 필드에 저장
  - 분실자에게 알림 발송 (사유 포함)

#### D-4. 수령 처리 (`/manager/pickups`)
- 수령 대기 목록 표시 (각 항목 카드 또는 테이블 행)
- 각 항목별 [수령 완료 처리] → 처리 화면
- **대리 수령 불가** 안내 문구 화면 상단 고정

**처리 화면 흐름**:
1. 분실자가 제시한 **신분증 이름** 입력
2. 분실자가 제시한 **연락처** 입력
3. 분실자가 제시한 **수령 코드** 입력
4. [검증] 버튼 → `POST /pickups/:pickupId/verify` → 응답 `allMatched: boolean` + 어떤 항목이 불일치인지 표시 (mismatches 배열)
5. 3가지 모두 일치할 때만 [수령 완료 확정] 버튼 활성
6. 불일치 시 명확한 에러: "이름이 일치하지 않습니다 / 연락처가 일치하지 않습니다 / 수령 코드가 일치하지 않습니다"
7. 확정 → 습득물 `수령완료`, 분실 신고 `찾기완료`, 매칭 비활성, 분실자에게 알림
8. 확정 후 안내: **"수령이 완료되었습니다. 분실자에게 물품을 전달한 뒤, '찾기완료 종료 처리' 페이지에서 종료 처리를 진행해주세요."**

#### D-5. 찾기완료 종료 처리 (`/manager/finalize`)
- 상태가 `찾기완료`인 분실 신고 목록 표시
- 컬럼: 신고자 / 물품명 / 수령 완료 일시 / 액션
- 안내: **"분실자에게 물품을 실물 전달 완료한 후 이 페이지에서 '종료 처리'를 눌러주세요. 종료된 신고는 통계 외에는 더 이상 노출되지 않습니다."**
- [종료 처리] 버튼 → 확인 모달 → `POST /reports/:reportId/finalize`
- 성공 시 해당 행 제거 + 토스트 알림

#### D-6. 폐기 검토 알림
- 헤더 알림 배지 + `/notifications`에서 `폐기검토필요` 타입으로 표시
- 알림 클릭 시 운영자 페이지 `/admin/unclaimed`로 이동 (관리자 권한으로 접근 가능)
- 또는 `/manager/items`에서 상태 필터를 '보관중'으로 두고 등록 일시 정렬

### E. 운영 관리자

> **참고**: 운영 관리자는 현재 우리팀(개발자)가 담당. 추후 서비스가 다른 학교로 확장되면 전체 학교 시스템을 관리하는 역할이 된다. 학교별 분리 같은 멀티 테넌트 기능은 v1에서는 다루지 않는다.

#### E-1. 대시보드 (`/admin`)
- 상태별 카드:
  - 분실 신고: 접수 / 매칭후보있음 / 찾기완료 / 종료
  - 습득물: 등록 / 보관중 / 수령대기 / 수령완료 / 폐기예정
- 간단한 막대 차트 (Recharts): 최근 30일 신고/등록 추이

#### E-2. 전체 분실 신고 (`/admin/reports`)
- 테이블 + 필터, 정렬, 검색
- 풀 정보 노출

#### E-3. 전체 습득물 (`/admin/items`)
- 위와 유사

#### E-4. 장기 미수령 (`/admin/unclaimed`)
- 상단에 안내: **"장기 미수령 = 폐기 검토 대기 (동의어)"**
- 컬럼: 물품명 / 카테고리 / 보관 위치 / 등록 일시 / 경과 일수 / 상태 / 액션
- [폐기예정으로 변경] 액션 (자동 전환 없음, 수동만)

#### E-5. 시스템 설정 (`/admin/settings`)
- 폼:
  - 장기 미수령 기준 (일) — 기본 30, 입력 가능
  - 수령 대기 자동 취소 기준 (일) — 기본 3
  - 매칭 시 날짜 근접 허용 (일) — 기본 7
- 저장 시 확인 모달 → PATCH
- 변경 후 안내: "전체 시스템 정책이 새 값으로 즉시 적용됩니다."

## 🧪 Mock 데이터 요구사항

`src/api/mock/data.ts`에 시드 데이터 정의, `src/api/mock/storage.ts`에 localStorage 연동:

- **사용자 10명**: 분실자 4, 습득자 3, 보관소 관리자 1, 운영자 1, 학생(분실자+습득자) 1
- **분실 신고 12건**: 접수 3, 매칭후보있음 4, 찾기완료 3, 종료 2
- **습득물 15건**: 등록 2, 보관중 6, 수령대기 2, 수령완료 3, 폐기예정 2
- **매칭 10건**: 활성 3, 확인요청중 2, 승인 1, 반려 2(반려 사유 포함), 비활성 2
- **수령 4건**: 수령대기 2, 수령완료 2
- **알림 시드**: 각 알림 타입별 1~2건씩, 미읽음과 읽음 혼재
- **시스템 설정**: 기본값 (30 / 3 / 7)

Mock은 localStorage 영속화. 헤더 우측에 (개발 모드에서만) **[Mock 초기화]** 버튼 노출 → seed로 리셋.

### Mock 핸들러 핵심 처리

`src/api/mock/handlers.ts`에는 단순히 데이터 반환만 하는 게 아니라 **자동 상태 전환과 마스킹**도 처리해야 한다.

**Mock에서 처리해야 할 자동화**:
- `createLostReport` 호출 시 → 매칭 후보 생성 시뮬레이션 → 카테고리·장소·날짜±N일 일치하는 습득물 찾아 `Match` 레코드 생성 + 분실 신고 상태를 `매칭후보있음`으로 자동 변경 + `매칭후보생성` 알림 생성
- `updateLostReport` 호출 시 → 활성 확인 요청 있으면 409 응답 → 없으면 기존 매칭 비활성화 + 재계산
- `deleteLostReport` 호출 시 → 활성 확인 요청 있으면 409 → 없으면 CASCADE 매칭 삭제
- `approveMatch` 호출 시 → Pickup 자동 생성, 수령 코드 6자리 랜덤, 습득물 → 수령대기, 알림 생성
- `rejectMatch` 호출 시 → 매칭 상태 `활성`으로 복귀, `rejectReason` 저장, 알림 생성
- `completePickup` 호출 시 → 습득물 수령완료, 분실 신고 찾기완료, 매칭 비활성
- `cancelPickup` 호출 시 → 습득물 보관중 복귀, 분실 신고 매칭후보있음 복귀, 매칭 활성 복귀, 알림 생성
- `finalizeReport` 호출 시 → 찾기완료인 분실 신고를 종료로 변경
- 비로그인 검색 호출 시 → 응답 데이터에서 신고자 이름·연락처·상세설명 마스킹 처리

이렇게 해야 Mock 모드에서도 **실제 시스템처럼 작동**하고 모든 흐름을 테스트할 수 있다.
